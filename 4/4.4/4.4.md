# 4.4. Módulo 4

# Caso de Uso1: Generación de Pedido
## Descripción del Caso de Uso: 
El distribuidor utiliza esta pantalla para crear un nuevo pedido en el sistema PRETUPS.

## Actor: Distribuidor

## Precondiciones:

- El distribuidor debe tener una cuenta activa en el sistema PRETUPS.

- El distribuidor debe tener crédito disponible para realizar pedidos.

## Postcondiciones:

- El pedido se genera y se envía al sistema para su validación.
- Se muestra un mensaje de confirmación si el pedido fue enviado correctamente.

## Flujo Principal:

| Nro |	Acción del Actor (AS IS)  |	Respuesta del Sistema (TO BE) |
|-----|------------------|-----------------------|
1 |	El distribuidor accede a la pantalla de generación de pedidos. |	El sistema carga automáticamente la pantalla de generación de pedidos y muestra el saldo de crédito disponible. | 
|2|El distribuidor selecciona los productos y cantidades.|El sistema muestra los productos y actualiza el resumen del pedido con el total del pedido.	
3	| El distribuidor confirma el pedido.	|El sistema valida que haya suficiente crédito disponible y envía el pedido para su verificación.|

## Requisitos:

- El sistema debe mostrar el saldo de crédito disponible.
- Debe permitir la selección de productos y cantidades.
- Confirmación del pedido al finalizar.

# Caso de Uso2: Verificación de Crédito y Resultado del Pedido


## Descripción del Caso de Uso: 
El sistema verifica automáticamente si el distribuidor tiene crédito suficiente para el pedido y muestra el resultado de la verificación.

## Actor: Distribuidor

## Precondiciones:

- El distribuidor debe haber enviado un pedido.

## Postcondiciones:

- Se muestra si el pedido fue aprobado o rechazado.
- Si es rechazado, se indica la razón.
- Si es aprobado, se genera una factura automáticamente.

## Flujo Principal:

Nro |	Acción del Actor (AS IS)	|Respuesta del Sistema (TO BE)|
 |-----|------------------|-----------------------|
1	|El distribuidor espera la verificación de crédito.|	El sistema verifica automáticamente el saldo de crédito disponible.|
2	|-	|El sistema valida si el crédito es suficiente para el pedido.|
3	|-	|Si el crédito es insuficiente, el sistema rechaza el pedido e informa al distribuidor del rechazo, especificando la razón.|
4	|-	|Si el crédito es suficiente, el sistema aprueba el pedido y genera una factura automáticamente en SAP.


## Requisitos:

- Verificación automática de crédito.
- Mensaje claro de rechazo o aprobación.

## Caso de Uso 3: Confirmación de Pedido Aprobado
## Descripción del Caso de Uso:
El distribuidor recibe una confirmación de que su pedido ha sido aprobado y puede ver los detalles de la factura generada.

## Actor(es):
- Distribuidor

## Precondiciones:

- El pedido debe haber sido aprobado.
Postcondiciones:

- El distribuidor puede descargar o visualizar la factura.
## Flujo Principal:

Nro	|Acción del Actor (AS IS)	|Respuesta del Sistema (TO BE)|
 |-----|------------------|-----------------------|
1	|El distribuidor revisa el estado de su pedido.	|El sistema muestra un mensaje de confirmación de que el pedido fue aprobado exitosamente.|
2	|-	|El sistema muestra los detalles de la factura generada automáticamente por SAP.|
3	|El distribuidor desea ver o descargar la factura.	|El sistema permite al distribuidor descargar la factura en formato PDF o visualizarla en pantalla.|

## Requisitos:

- Visualización de la factura generada.
- Opción para descargar la factura.

# Caso de Uso 4: Verificación de Importes entre SAP y PRETUPS
## Descripción del Caso de Uso:
El sistema compara los importes de la factura generada en SAP con los datos en PRETUPS y verifica que coincidan.

## Actor(es):
Sistema (automático)

## Precondiciones:

- Una factura debe haber sido generada por un pedido aprobado.
## Postcondiciones:

- Se confirma que los importes coinciden entre SAP y PRETUPS.
- Si hay discrepancias, se envía una notificación para revisión.
## Flujo Principal:

Nro	|Acción del Actor (AS IS)	|Respuesta del Sistema (TO BE)|
|-----|------------------|-----------------------|
1	|El sistema espera generar y verificar la factura.|	El sistema compara automáticamente los importes de la factura generada en SAP con los datos en PRETUPS.|
2	|-	|Si los importes coinciden, el sistema confirma la verificación exitosa y continúa el proceso.|
3	|-	|Si hay discrepancias, el sistema muestra un mensaje de error y envía una notificación al equipo responsable para revisión.|
## Requisitos:

- Comparación automática de importes.
- Notificación en caso de discrepancias.


Notificación Automática al Distribuidor
Descripción del Caso de Uso:
El distribuidor recibe una notificación automática sobre el estado de su pedido y la generación de la factura.

Actor(es):
Distribuidor, Sistema (automático)

Precondiciones:

El pedido debe haber sido procesado (aprobado o rechazado).
Postcondiciones:

El distribuidor recibe una notificación con el estado del pedido y detalles de la factura, si aplica.
Flujo Principal:

Nro	Acción del Actor (AS IS)	Respuesta del Sistema (TO BE)
1	El distribuidor espera el estado de su pedido.	El sistema envía automáticamente una notificación al distribuidor sobre el estado del pedido.
2	-	El distribuidor recibe una notificación con el estado del pedido y los detalles de la factura.
3	El distribuidor revisa la notificación.	El sistema permite al distribuidor revisar el estado del pedido en la pantalla de notificaciones.
Requisitos:

Notificación automática para informar sobre el estado del pedido.
6. Pantalla de Reporte Diario de Transacciones
Descripción del Caso de Uso:
Se genera un reporte diario con todas las transacciones realizadas, que puede ser visualizado y exportado por los administradores.

Actor(es):
Administrador, Jefe de Área

Precondiciones:

El sistema debe haber procesado transacciones durante el día.
Postcondiciones:

El reporte diario está disponible para consulta y exportación.
Flujo Principal:

Nro	|Acción del Actor (AS IS)	|Respuesta del Sistema (TO BE)|
|-----|------------------|-----------------------|
1	|El administrador accede a la pantalla de reportes diarios.	|El sistema genera automáticamente un reporte diario con todas las transacciones realizadas.|
2	|El administrador revisa el reporte.	|El sistema permite al administrador visualizar un resumen de las transacciones.|
3	|El administrador desea exportar el reporte.	|El sistema permite al administrador exportar el reporte en formatos como PDF o Excel.|

## Requisitos:

- Generación automática de reportes diarios.
- Opciones de exportación de los reportes.
